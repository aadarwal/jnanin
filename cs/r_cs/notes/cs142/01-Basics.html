Introduction¶

Between last quarter and this quarter, we have seen data structures that allow us to model things in the real world:

  • Lists model lists, or any ordered information
  • Trees model things that are hierarchical, like people within organizations or a taxonomy of all known organisms
  • Dictionaries model associations between short identifiers and broader information about the identified entity, such as address books, language dictionaries, and so on

It is hard to imagine trying to implement a model of any of these real-world things without the corresponding data structure.

These data structures are also useful for computational purposes that do not involve modeling. For instance, a dictionary is a useful tool for keeping track of counts as we analyze the statistics of data. And imposing an artificial hierarchy on data that is not intrinsically hierarchical can provide more efficient operations on it (BSTs).

We now introduce a new data structure that lets us model another kind of thing in the real world, something not effectively represented by these other data structures. Specifically, graphs allow us to model entities and the relationships between them.

Terminology and Examples¶

In a graph, a vertex (plural, vertices), sometimes casually called node(s), model entities. Examples:

  • people
  • cities
  • patents
  • ...

Edges between pairs of vertices model entities that have relationships. Examples:

  • people who know each other
  • cities linked by a direct flight
  • patents that cite other patents
  • ...

Graphs can be directed or undirected:

  • in an undirected graph, all relationships are reciprocated. The graph above is an example of an undirected graph; for example, if the vertices represented cities and the edges represented the relationship "is reachable by road from", then the edge between A and B tells me that city A is reachable from B, and that the reciprocal relationship (B is reachable from A) is also true.
  • in a directed graph, relationships can be one-sided. The graph below is an example of a directed graph. Suppose the vertices still represent cities, but the edges now represent "there exists a flight from one vertex to another". In this graph, there is a flight going from A to D, but not from D to A (if I wanted to get from D to A, I would have to fly from D to C, and then from C to A). Notice how this graph doesn't preclude some vertices having reciprocal edges: it is still possible to fly from A to B and from B to A.

Graphs can be weighted or unweighted:

  • in a weighted graph, numerical values are placed on edges to measure the strength of the relationship.
  • in an unweighted graph, all relationships are of equal strength or the idea of strength is irrelevant to the problem at hand. All the graphs shown above are unweighted.

Following from the previous example (where the edges represent flights between cities), the value associated with each edge could represent the price of a plane ticket (but, depending on the problem, we could choose other weights: the distance between cities, the amount of fuel required by that flight, etc.)

Storing Graphs¶

The two most-commonly-used ways of representing a graph in a data structure are adjacency lists and adjacency matrices.

Adjacency Lists¶

In this representation, each origin vertex has an associated list of other vertices to which it has an edge.

In an unweighted graph, this is just a list of destination edges. In a weighted graph, the entries in this list are conceptually tuples of destination and weight.

Depending upon whether a graph is directed or undirected, reciprocal edges are or are not guaranteed to appear in the lists.

Python Objects for Adjacency Lists¶

Rather than having lists of node identifiers, in a Python implementation, we might have classes for vertices, with one of their attributes a list of the other vertex objects to which a given vertex has edges.

In [8]:
from typing import Optional

class Vertex:
    
    name: str
    edges_to: set["Vertex"]

    def __init__(self, name: str):
        self.name = name
        self.edges_to = set()
    
    def add_edge_to(self, dest: "Vertex") -> None:
        self.edges_to.add(dest)
        
    def has_edge_to(self, dest: "Vertex") -> bool:
        return dest in self.edges_to
        
class Graph:
    
    vertices: dict[str, Vertex]

    def __init__(self):
        self.vertices = {}
    
    def add_vertex(self, vertex: Vertex) -> None:
        self.vertices[vertex.name] = vertex
        
    def get_vertex(self, name: str) -> Optional[Vertex]:
        return self.vertices.get(name)

Note that casually we might refer to adjacency "lists," but perhaps adjacency sets would be more appropriate (and efficient).

class Vertex:
    name: str
    edges_to: dict[str, tuple["Vertex", int]]

    def __init__(self, name: str):
        self.name = name
        self.edges_to = {}

    def add_edge_to(self, dest: "Vertex", weight: int) -> None:
        self.edges_to[dest.name] = (dest, weight)

    def has_edge_to(self, dest: "Vertex") -> bool:
        return dest.name in self.edges_to

    def weight_to_dest(self, dest: "Vertex") -> int:
        if dest.name in self.edges_to:
            return self.edges_to[dest.name][1]
        return 0


class Graph:
    vertices: dict[str, Vertex]

    def __init__(self):
        self.vertices = {}

    def add_vertex(self, vertex: Vertex) -> None:
        self.vertices[vertex.name] = vertex

    def get_vertex(self, name: str) -> Optional[Vertex]:
        return self.vertices.get(name)

Adjacency Matrices¶

In this course we will use an adjacency list representation when working with graphs, but another common representation for graphs is with an adjacency matrix, which typically performs better than adjacency lists for very dense graphs (i.e., graphs where almost all nodes are connected to each other). We explain below how to represent a graph using a matrix, but won't go into the implementation of this representation.

When using an adjacency matrix representation, we will use a two-dimensional matrix to specify whether there is an edge between one vertex and another.

More specifically, given an unweighted graph with $N$ vertices, we have an $N \times N$ matrix where each location $(i, j)$ is set to 1 if there is an edge going from $i$ to $j$ and 0 otherwise. Notice how this means that the vertices are numbered from $0$ to $N-1$ (although we can then associate string labels to each number).

For example, given this undirected graph:

The adjacency matrix would look like this:

      A  B  C  D  E  F
A  [[ 0, 1, 1, 1, 0, 0],
B   [ 1, 0, 0, 1, 0, 1],
C   [ 1, 0, 0, 1, 1, 0],
D   [ 1, 1, 1, 0, 0, 1],
E   [ 0, 0, 1, 0, 0, 0],
F   [ 0, 1, 0, 1, 0, 0]]

Notice how, for an undirected graph, the matrix will always be symmetrical (because an edge from $i$ to $j$ always implies an edge from $j$ to $i$).

Given a directed graph:

The adjacency matrix would look like this:

      A  B  C  D  E  F
A  [[ 0, 1, 0, 1, 1, 0],
B   [ 1, 0, 0, 1, 0, 1],
C   [ 1, 0, 0, 0, 1, 0],
D   [ 0, 0, 1, 0, 0, 0],
E   [ 0, 0, 1, 0, 0, 0],
F   [ 0, 0, 1, 1, 0, 0]]

In a weighted graph, the adjacency matrix contains the weights of the edges (or 0 if there is no edge between two vertices).
