The most important technique for achieving deep modules is information hiding. Each module encapsulates a few pieces of knowledge (data structures, algorithms, etc.) that represent design decisions. This knowledge is embedded in the implementation, but does not appear in the interface, so it is not visible to other modules.

The most common type of information hidden in a module is what we informally refer to as “implementation details”. For example:

  • The implementation of a “dictionary” (could be a tree, a hash table, etc.)
  • The details of a network protocol
  • How to parse a given file format

The opposite of information hiding is information leakage. Information leakage occurs when a design decision is reflected in multiple modules, creating a dependency.

Example: Exposing internal data structures¶

Consider the following classes to model courses, and students who register for a specific course offering, at a university. For simplicity, let's assume a course can only be offered once per year:

Example: File Formats¶

Suppose we are implementing some chess software, that uses a text-based file format to represent chess boards. For example, this chess board:

Would be represented as follows:

R_BK_B_R
PP_NQ_P_
__P_PNbP
________
___p_b__
_____n__
ppp__ppp
r__q_rk_

Each line represents a row of the chessboard. Empty spots are represented by an underscore character, black pieces are represented with uppercase letters, and white pieces are represented with lowercase letters:

  • p/P: Pawn
  • r/R: Rook
  • b/B: Bishop
  • n/N: Knight
  • q/Q: Queen
  • k/K: King

When writing our code, we could be tempted to do something like this:

In [11]:
class Piece:
    char: str
    
    def __init__(self, char: str):
        self.char = char
        
    def is_white(self) -> bool:
        return self.char.islower()
    
    def is_black(self) -> bool:
        return self.char.isupper()
    
    def __str__(self) -> str:
        if self.char == "p" or self.char == "P":
            return "Pawn"
        elif self.char == "r" or self.char == "R":
            return "Rook"
        # etc.

This code is easy and convenient: we can iterate over the contents of the file and just pass the character in the file to the Piece constructor:

In [12]:
p1 = Piece("R")
p2 = Piece("p")
In [13]:
str(p1)
Out[13]:
'Rook'
In [14]:
p2.is_white()
Out[14]:
True
In [15]:
p2.is_black()
Out[15]:
False

Q: What issues do you see with this design?

A: Suppose the file format changes; we either need to scrap our Piece class entirely or (even worse) add more code to our software to convert form the new file format to an old file format that is now just being used internally in our code.

Even if the file format doesn't change, what if we want to support multiple file formats? Once again, we'd have to add code to convert from other file format to an arbitrarily chosen format.

This design also has a high cognitive load: anyone using our Piece class would need to be familiar with the one-character representation for these chess pieces.

This is a different kind of information leakage: we're leaking information about a file format we are using. We will need to write code to deal with that file format, but information about that file format shouldn't leak outside of the code that reads in the files (and into the implementation of the Piece class itself)

A better solution (but which can still be improved) would be something like this:

In [20]:
class Piece:
    
    PAWN = 0
    ROOK = 1
    BISHOP = 2
    KNIGHT = 3
    QUEEN = 4
    KING = 5
    
    WHITE = 0
    BLACK = 1
    
    piece_type: int
    color: int
    
    def __init__(self, piece_type: int, color: int):
        self.type = piece_type
        self.color = color

Creating pieces now looks like this:

In [21]:
p1 = Piece(Piece.PAWN, Piece.WHITE)

The above style is pretty common, and your likely to encounter it in the wild, but a better approach would be to use "enums" ("enumerations"). Enums allow us to easily create a new type with a limited set of values. For example, the PieceColor type below can only take two values: BLACK or WHITE.

In [11]:
from enum import Enum

class PieceColor(Enum):
    BLACK = 0
    WHITE = 1

class PieceType(Enum):
    PAWN = 0
    ROOK = 1
    BISHOP = 2
    KNIGHT = 3
    QUEEN = 4
    KING = 5

class Piece:
       
    piece_type: PieceType
    color: PieceColor
        
    def __init__(self, piece_type: PieceType, color: PieceColor):
        self.type = piece_type
        self.color = color

Creating pieces looks very similar:

In [12]:
p1 = Piece(PieceType.PAWN, PieceColor.WHITE)

The main takeaway in both approaches is that we've decoupled the design of our Piece class from any particular file format, and the details of how to construct and use the Piece class don't require knowledge of any external file format.

Note: There is a more compact way of declaring enums that you'll see in some of the project code. For example, the enums shown above can be re-written like this:

In [13]:
PieceColor = Enum("PieceColor", ["BLACK", "WHITE"])
PieceType = Enum("PieceType", ["PAWN", "ROOK", "BISHOP",
                               "KNIGHT", "QUEEN", "KING"])
