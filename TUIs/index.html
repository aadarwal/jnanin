TUI frameworks & approaches

Notes on what makes a good terminal UI framework, what the real tradeoffs are between Go, Rust, and Python in this space, and which one to reach for when. This page grows as I evaluate more options.

Started 2026-05-27 · Currently shipping: artha (Go + Bubble Tea)

TL;DR

The rule of thumb

If your TUI could plausibly fit on a single screen with keyboard navigation, Bubble Tea (Go). If it's a multi-pane, mouse-driven dashboard with scrollable history and animations, Textual (Python). If you'd reach for Rust regardless of the project, Ratatui (Rust).

For most personal-tool TUIs — the kind you symlink into ~/.local/bin/ and run a hundred times a week — Go is the right default. Distribution alone wins it.

The three contenders

These are the three frameworks worth taking seriously today. Others exist (Ink for Node, ncurses for C, FTXUI for C++) but none match these on ecosystem and ergonomics.

Bubble Tea — Go

Mental model

The Elm Architecture, ported faithfully. Your whole app is three things: a Model (all state), an Update(msg, model) → (model, cmd) function (every state change), and a View(model) → string (renders the full screen each frame). Messages flow through one channel. Async work is wrapped in a Cmd that runs in a goroutine and emits a message when done.

func (m model) Update(msg tea.Msg) (tea.Model, tea.Cmd) {
    switch msg := msg.(type) {
    case tickMsg:     return m, fetchPrices    // async cmd
    case pricesMsg:   m.prices = msg.data; return m, nil
    case tea.KeyMsg:  // handle keys
    }
    return m, nil
}

Strengths

  • The rigidity is the feature. One way to do things → apps stay clean as they grow.
  • Single static binary. Cross-compile from Mac to Linux/Windows with one env var.
  • Goroutines map perfectly onto TUI work — background fetches without blocking the UI.
  • The Charm ecosystem (Bubbles, Lipgloss, Huh, Glamour, Wish, VHS) is the gold standard.

Weaknesses

  • Mouse support, scrollable panes, and modals are DIY (improving fast, still not Textual-level).
  • Re-rendering the full screen each frame is fine, but if you want partial updates you'll fight the framework.
  • Go's verbosity shows up in long switch msg.(type) blocks.

Ratatui — Rust

Mental model

Immediate-mode. You own the event loop. Each frame you call terminal.draw(|f| { ... }) and rebuild the widget tree from scratch out of your current state. Widgets are stateless values constructed each frame. There is no framework-imposed architecture — you decide where state lives and how it changes.

loop {
    terminal.draw(|f| {
        let block = Block::default().title("Prices").borders(Borders::ALL);
        f.render_widget(Paragraph::new(state.text()).block(block), f.size());
    })?;
    if let Event::Key(key) = event::read()? { /* mutate state */ }
}

Strengths

  • Total control: layout, render timing, allocations. Nothing happens behind your back.
  • Truly fast — sub-5ms startup, ~3MB memory. The only one where perf is a reason.
  • No GC pauses. Matters for animation-heavy or live-data TUIs (it won't for most of yours).
  • Single static binary, like Go.

Weaknesses

  • More boilerplate. A 50-line Bubble Tea app is 150 lines of Ratatui.
  • Slow compile times slow iteration.
  • No imposed architecture means your second app may look nothing like your first.
  • Async (tokio) is opt-in and adds a layer of complexity.

Textual — Python

Mental model

A DOM tree of Widget objects with parent/child relationships, styled by a real CSS file (.tcss). State lives in reactive attributes that auto-trigger re-renders when they change. Messages bubble up the widget tree, like DOM events. Built on top of Rich (the rendering library).

class Portfolio(App):
    CSS_PATH = "portfolio.tcss"
    prices = reactive({})

    def compose(self) -> ComposeResult:
        yield Header()
        yield DataTable(id="holdings")
        yield Footer()

    def watch_prices(self, new):
        self.query_one(DataTable).update(new)

Strengths

  • Mouse, scrollable containers, modals, animations, a command palette — all standard.
  • Real CSS file with live reload in dev mode (textual run --dev).
  • Can also serve your TUI as a web app via textual-web. Unique to this stack.
  • Fastest iteration loop — no compile, hot CSS reload.

Weaknesses

  • Distribution is the wart. pip install, pipx, or a fat PyInstaller bundle. None as clean as go build.
  • Startup latency: 200–500ms (importing Textual + Rich). Noticeable for tools you launch often.
  • ~50MB resident. Fine for dashboards; heavy for utilities.
  • Async-first (asyncio) — adds cognitive overhead if you're not already living in it.

Decision guide

If
A CLI-style tool you'll install with one binary and run hundreds of times
Bubble Tea. Distribution + startup + Charm ecosystem. This is where artha lives.
If
A multi-pane mouseable dashboard with scrollable history and live updates
Textual. The widget primitives and CSS make this 5× faster to build. Accept that users install via pipx.
If
You want to learn Rust, or your TUI is part of a bigger Rust project
Ratatui. Don't pick it for TUIs alone — pick it if Rust is the broader investment.
If
You want to share the TUI over SSH or expose it to non-terminal users
Bubble Tea + Wish (SSH server hosting a TUI), or Textual + textual-web (TUI rendered in a browser). Different vibes — try both.
